## What we ran

```bash
python3 scripts/aft_extract_training_data.py extract --max-prs 500 --output data/aft/pr_triage_corpus.jsonl
python3 scripts/aft_extract_training_data.py split   # 80/20-ish, balanced minority class
python3 scripts/aft_harness.py --frontier-dry-run \
    --advocate-cmd "bin/aft-advocate --backend stub"
```

The `--frontier-dry-run` flag means the `frontier_rules` condition used the
in-process deterministic heuristic, **not** a real frontier API call. The
`--backend stub` flag means the `local_advocate` condition used the shim's
heuristic stub, **not** a trained model. This is a **baseline**, not the
empirical answer to the AFT hypothesis. We are establishing that the
plumbing works end-to-end on real data before any model is trained.

## Corpus stats

- 556 PRs extracted across merged/closed/open
- Class balance: `merged_fast` Ōēł 500, `closed_no_merge` Ōēł 56, `open_aged` = 0

## Substantive finding: the `open_aged` class is empty in current data

In the current operating state of the repo, **no open PR is more than 14
days old**. The aggressive merge cadence of the past two weeks
(28+ bounded-drain merges, see `docs/status/QUEUE_DRAIN_FINAL_claude-F1CD271D.md`)
has cleared the historical "open and stalled" backlog. This is not a bug in
the extractor ŌĆö it is a true description of the queue.

Implications for the experiment design:

1. The PR triage task is effectively a **2-class problem** today
   (`merged_fast` vs `closed_no_merge`), not 3-class as the spec assumed.
2. Heuristics that *default to* `open_aged` (such as the current
   `FrontierRules` dry-run heuristic) will perform *worse* than a
   majority-class baseline on this data.
3. The label space stays 3-class in code, because the moment merge cadence
   slows down, `open_aged` will re-emerge as a non-empty class. We do not
   want to hard-code a 2-class assumption that will break silently.

## What this baseline tells us

1. **Plumbing works end-to-end.** `gh` extract ŌåÆ split ŌåÆ harness ŌåÆ
   stub-shim round-trip ŌåÆ summary report all complete without error against
   real data. The Brier scorer, McNemar test, and Bonferroni correction
   all produce sensible numbers.
2. **The default `frontier_rules` heuristic is mis-tuned for the current
   data distribution.** It predicts `open_aged` 50% of the time on a
   holdout where that class is empty. When we wire a real frontier API in,
   the operator rules in the prompt should be updated to reflect the
   2-class reality, *and* the prompt should bias toward majority class on
   ambiguous inputs ŌĆö the current rule sheet does the opposite.
3. **The stub `local_advocate` is intentionally worse than a real
   advocate would be**, and the report correctly does not mark it as an
   empirical signal (no `_stubbed` suffix because the shim ran
   successfully; the stub is the *configured* backend, which the operator
   chose explicitly). This is fine for a plumbing run; we will not draw
   conclusions from these numbers about the advocate hypothesis.

## What changes before the real H1/H2 run

1. **Refresh the rule sheet.** Update `FrontierRules.OPERATOR_RULES` and
   the shim's `SYSTEM_PROMPT` to reflect the 2-class reality and to bias
   toward the majority class when seeds are uninformative. *Or* (cleaner)
   regenerate the corpus with a longer time window so `open_aged` is
   non-empty.
2. **Either widen the corpus or shrink the label space.** Two options:
   - Pull PRs from before the current merge cadence began (Mar-Apr 2026)
     where many PRs were genuinely open >14 days.
   - Officially make the v0.1 task 2-class and re-spec; this is allowed
     by `docs/specs/LOCAL_ADVOCATE_TRAINING_PIPELINE.md`'s "Sample size
     sanity" subsection.
3. **Wire the real frontier.** Run with `aragora ask` or `claude --print`
   as the `FrontierRules` backend and re-record the row. Then the
   `frontier_rules` row in this table becomes an empirical comparison
   target rather than a plumbing check.
4. **Train the Tier A LoRA adapter.** Per
   `docs/specs/LOCAL_ADVOCATE_TRAINING_PIPELINE.md` step 3, on the
   `pr_triage_train.jsonl` corpus, via MLX. Then re-run with
   `--advocate-cmd "bin/aft-advocate --backend mlx --model <...> --adapter <...>"`
   and write `docs/status/AFT_RESULT_v0.1.md`.
