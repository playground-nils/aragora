## Thesis

AI coding agents are increasing PR volume faster than engineering teams can
review it. The bottleneck is shifting from code generation to trustworthy
judgment.

Aragora should not respond by removing the human merge gate. Aragora should
turn review into its own product surface: a **heterogeneous PR intelligence
brief** that compresses consequential code review into a receipt-backed,
high-signal settlement packet.

The correct framing is not "another PR review bot." The correct framing is:

- a batched decision brief for software change
- generated by multiple heterogeneous agents with explicit dissent
- grounded in repo evidence, CI, validation, and risk signals
- settled by a human on the exact reviewed head SHA

This is the software equivalent of an executive intelligence brief:
prioritized, auditable, skim-first, and expandable on demand.

## Why Now

Three repo truths make this timely:

1. Aragora already has a batched settlement design in
   [2026-04-19-batched-pr-review-triage.md](2026-04-19-batched-pr-review-triage.md).
2. The current queue has demonstrated that the machine-preparation layer can
   recover stale branches, generate packets, and keep PRs review-ready.
3. The remaining bottleneck is human review throughput, not raw code-writing
   capability.

This means the next wedge is not "more autonomous merging." It is "better
human settlement on more machine-prepared work."

## Market Signal

Recent product releases and industry coverage are converging on the same
problem statement: AI can generate, scan, and submit code faster than small
teams can review and absorb it.

That validates the direction, but it also sharpens the differentiation.
Homogeneous review agents and comment-heavy review bots will become common.
Aragora should compete on:

- heterogeneous disagreement instead of one-model certainty
- settlement packets instead of comment floods
- receipts and provenance instead of ephemeral output
- preserving the human merge gate while reducing review time

The product should assume that "AI review exists" becomes table stakes. The
wedge is trustworthy settlement, not merely automated detection.

## Category

**Receipt-backed heterogeneous review briefing for consequential code changes**

That category is intentionally narrower and more defensible than:

- faster PR review bot
- AI code reviewer
- GitHub add-on
- autonomous merge agent

The buyer promise is:

> Give me one truthful brief that tells me what matters about this PR, where
> the models disagree, what evidence supports the recommendation, and whether I
> can settle it safely in under two minutes.

## Problem Statement

Current review surfaces fail in one of four ways:

1. they optimize for comments instead of decisions
2. they are homogeneous and hide disagreement behind one model voice
3. they generate findings without durable provenance or receipt trails
4. they assume the human should read the diff first and the summary second

Aragora should invert that workflow:

1. read the brief first
2. expand only the risky or dissent-heavy regions
3. settle on the exact head SHA
4. route rejected work into a bounded repair lane with the rejection reason

## Product Shape

### Inputs

- PR metadata and changed-file list
- live required-check state
- diff stats and touched subsystem tags
- machine review findings from existing `review-pr`
- validation claims extracted from PR body, artifacts, or receipts
- historical receipts for the same subsystem when relevant

### Core Deliberation

Run a short bounded review debate with 3-5 heterogeneous agents.

Suggested roles:

- `logic_reviewer`: behavioural correctness and regression risk
- `security_reviewer`: auth, secrets, privilege, unsafe surface changes
- `maintainability_reviewer`: architecture drift, hidden coupling, repair cost
- `skeptic`: argues the brief is overstating certainty or missing counterevidence
- `synthesizer`: produces final settlement packet and dissent summary

The important property is heterogeneity, not just agent count. The system
should prefer materially different models or model families when budget allows.

### Outputs

Each PR gets one primary brief:

- top-line recommendation
  - `approve_candidate`
  - `needs_human_attention`
  - `repair_first`
- one-sentence rationale
- risk tier
- touched subsystems
- validation summary
- top 3 findings
- explicit dissent block
- cost used / budget remaining
- exact head SHA and packet SHA

Each brief also gets a structured receipt:

- agent roster
- evidence references
- model-level findings
- confidence / disagreement measures
- settlement linkage

## UI Shape

The right interface is not "show every comment." It is a **briefing surface**.

Minimum viable UI:

- priority-ranked PR cards
- red / yellow / green settlement recommendation
- one-paragraph executive summary
- expandable dissent panel
- expandable evidence / validation panel
- changed-files drill-down
- one-click actions:
  - approve
  - request changes
  - defer
  - open diff

The mental model should be:

- brief first
- evidence second
- raw diff third

## Differentiation Against Homogeneous Review

If competitors provide one-model or same-family multi-agent review, Aragora's
real wedge is:

- heterogeneous disagreement, not simulated plurality
- receipts and provenance, not ephemeral comments
- settlement gating, not just issue spotting
- bounded repair loops after rejection
- human-control-by-design rather than optional human override

The message is:

> Most tools help AI write more code. Aragora helps humans safely decide what
> to merge when AI has written too much code too quickly.

## Safety Boundary

This surface should preserve the current approved boundary:

- machine review is advisory
- human settlement remains mandatory for merge
- repair re-dispatch stays bounded and auditable
- no bot-approves-bot path

That boundary is product truth, not temporary caution.

## Architecture Reuse

The design should reuse existing repo assets:

- `aragora/cli/commands/review_pr.py`
- `aragora/cli/commands/review_queue.py`
- `aragora/swarm/merge_arbiter.py`
- `aragora/debate/`
- receipt / settlement / ledger infrastructure
- `aragora/live/` app surfaces for settlement and operator views

The missing work is mostly composition and presentation:

- `PRReviewProtocol` over heterogeneous agents
- brief-oriented receipt schema
- cost-aware review-policy controls
- packet/brief UI for settlement

## MVP

MVP should stay bounded:

1. run only on Aragora's own PRs
2. use the existing review queue as the primary source
3. generate one brief per PR on demand or when a PR enters `ready_now`
4. expose CLI and JSON artifact first
5. add UI once packet quality is stable

## Success Criteria

### Operator outcomes

- one human can settle 20-50 small PRs in 10-15 minutes
- fewer diff-open actions are required for low-risk PRs
- fewer unsafe approvals occur due to hidden disagreement

### Product outcomes

- packet disagreement correlates with later review friction or repair need
- settlement time falls without increasing post-merge regression rate
- review packets become a reusable surface for prospects and design partners

### GTM outcomes

- Aragora can be shown as "the decision brief for code change"
- the surface is intelligible to non-specialist operators
- the product story is visibly different from "PR review bot"

## Non-Goals

- replacing GitHub as a source host
- replacing static analysis or normal CI
- reviewing every trivial PR with full debate depth
- auto-merging without explicit human settlement
- claiming that AI review removes engineering ownership

## Sequencing

1. stabilize the current batched settlement path
2. add heterogeneous review packet generation
3. add receipt schema extensions for dissent and evidence
4. add the briefing UI
5. add bounded repair re-dispatch after rejection

This should be treated as a productization layer on top of the existing
batched-triage path, not a separate program.
