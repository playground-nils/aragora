# Swarm substrate PR review — 2026-04-17

Reviewer: Claude Opus 4.7 (independent second-opinion). All five PRs were authored
and merged via Codex-based automation within ~4 hours of each other on 2026-04-17;
this is a cross-family review for regressions and hidden coupling that same-family
review tends to miss.

Scope reviewed (in priority order):

1. #6071 — `fix(swarm): prevent decomposer queue leakage and re-attempt spam`
2. #6072 — `fix(swarm): parse launchd probe top-level state`
3. #6070 — `test(proof-first): cover benchmark publication gate`
4. #6053 — `fix(swarm): stabilize lane telemetry db path`
5. #6054 — `chore(governance): reset contract drift ratchet date`

## Executive summary

- **Overall risk: LOW–MEDIUM.** All five PRs do what they claim and ship
  reasonable regression tests. The substrate is meaningfully safer after this
  batch. One PR (#6071) introduces a narrow, *secondary* regression in the
  decomposer's duplicate-child coverage check that is likely harmless today but
  will bite if the idempotency guard is ever bypassed. Another (#6054) is a
  policy-legal but governance-hazardous ratchet reset that should not become a
  habit.
- **PR with biggest concern: #6071** — the idempotency guard narrows re-entry
  spam, but the fix simultaneously removes the `boss-ready` label from
  decomposer children, which silently weakens the same function's existing
  `gh issue list --label boss-ready` duplicate-coverage search
  (`aragora/swarm/boss_loop.py:1497-1509`). The two changes interact but neither
  change mentions the interaction.
- **Immediate follow-up PRs recommended:** no critical follow-ups. Two
  medium-term items:
  1. Replace the decomposer's `--label boss-ready` duplicate-coverage query
     with a title/lineage-based query (`[from #N]` regex) so it still finds
     unlabeled children.
  2. Record a policy that the contract-drift ratchet reset in #6054 is a
     one-time grace, with a deadline for when the 655-item baseline also
     moves. Otherwise every future miss will be "resolved" by resetting the
     start date.

### #6071 — `fix(swarm): prevent decomposer queue leakage and re-attempt spam`

**Scope:** Two changes in `aragora/swarm/boss_loop.py::_auto_decompose_stuck_issue`:

1. Early-return when the issue already carries `boss-stuck`
   (`aragora/swarm/boss_loop.py:1397-1407`).
2. Stop passing `--label boss-ready` when creating decomposer sub-issues
   (`aragora/swarm/boss_loop.py:1651-1670` — note the comment and the missing
   `--label boss-ready` args that were in the old form at the same site). The
   parent-issue comment is correspondingly reworded.

Plus test-file reshape (`tests/swarm/test_boss_decomposition_depth.py`): fix
the pre-existing `_label_boss_stuck.assert_not_called()` assertion at shallow
depths (the code path *always* calls `_label_boss_stuck` at function exit), and
add two regression tests (`test_auto_decompose_skips_when_issue_already_boss_stuck`,
`test_auto_decompose_does_not_label_children_boss_ready`).

**Correctness: PARTIAL (fix works, carries a new secondary regression).**

The primary claim — "four 'exhausted 3 attempts' comments in 13 minutes on
#5894 stop happening" — is genuinely addressed:

- `_already_maxed_issue_numbers` calls `_auto_decompose_stuck_issue` whenever
  a count is *equal* to `max_retries_per_issue`
  (`aragora/swarm/boss_loop.py:917`). Because the comparison is `==`, the call
  recurs every iteration where the counter sits at exactly the max. Iteration
  N+1 refetches `issues` and thus sees the `boss-stuck` label applied by
  iteration N, and the guard short-circuits. The 5+ minute spacing in the field
  observation is consistent with iteration cadence. Field fix: holds.
- The `issue.labels` check uses fetched labels, so the guard is
  best-effort against *concurrent* boss-loop instances — two processes that
  fetch issues before either labels will both pass the guard. The PR body
  acknowledges this as "stale feed caches or concurrent boss-loops"; the
  mitigation is probabilistic, not a lock. For the single-daemon case on macOS
  (one LaunchAgent) this is fine; if multiple hosts ever co-run the boss loop,
  a per-issue lease or `gh label list --json` live read is required.

**Queue-leakage change is correct** per `docs/status/NEXT_STEPS_CANONICAL.md`
(only `CS-01..03` carry `boss-ready` in this tranche).

**Secondary regression** — the decomposer's own duplicate-coverage lookup still
queries `gh issue list --label boss-ready --state open`
(`aragora/swarm/boss_loop.py:1497-1509`) and feeds the returned issues into
`existing_titles` / `existing_decomposition_signatures` used by
`_decomposition_candidate_is_covered`. After this PR, decomposer-produced
children no longer have `boss-ready`, so they *cannot* appear in this list.
If `_auto_decompose_stuck_issue` is ever re-entered on the same parent (human
removes `boss-stuck`, or the concurrent-loop race described above), the
function will not recognise previously-created children and can duplicate them.
Today the idempotency guard masks this; the coupling should be made explicit
before the guard is ever weakened.

**Regression risk:** Low for today's operating model (single LaunchAgent,
canonical `boss-ready` tranche empty beyond CS-01..03). Medium the moment
anyone promotes decomposer children to `boss-ready` without also updating
duplicate detection, or re-opens a stuck issue.

**Test coverage adequacy:** Good for the two stated bugs, incomplete for the
full function. What's exercised:

- Idempotency guard short-circuits (`tests/swarm/test_boss_decomposition_depth.py:71-105`).
- No `--label boss-ready` passed to `gh issue create`
  (`tests/swarm/test_boss_decomposition_depth.py:106-160`).
- Fixed the pre-existing wrong assertion that `_label_boss_stuck` is not
  called at shallow depths; it *is* called at function exit. The new
  assertion (`label_stuck.assert_called_once()`) is correct and strictly
  stricter than the old one.

What is not exercised:

- Duplicate-detection when a prior decomposition's children have *no* labels
  (the regression described above). A targeted test should assert that when
  `gh issue list --label boss-ready` returns empty but the parent already has
  sibling children discoverable via title/lineage, the second run does not
  create duplicates.
- The concurrent-loop race. Hard to test deterministically but could be
  partially covered by asserting the guard falls back to a live GitHub label
  lookup — which it currently doesn't.

**Issues found:**

1. `aragora/swarm/boss_loop.py:1497-1509` — `existing_titles` /
   `existing_decomposition_signatures` are sourced from
   `gh issue list --label boss-ready --state open`, which can no longer see
   this function's own outputs. **Recommended follow-up**: add a second query
   keyed on `[from #<parent>]` title prefix or search the full open-issue set
   for the lineage marker.
2. `aragora/swarm/boss_loop.py:1384-1407` — the idempotency guard is the
   *only* gate against re-entry spam. If a human removes `boss-stuck` to
   retry, and `_issue_attempt_counts[issue.number]` is still `== max_retries`
   (it is not cleared anywhere I can find), the function runs again with no
   new attempts recorded — it will re-run every subsequent iteration because
   the counter never moves. The first pass will re-label `boss-stuck` at exit,
   so only one duplicate comment happens; still worth explicitly clearing or
   decrementing the counter on `boss-stuck` removal, or keying re-entry off
   GitHub's *live* label list rather than stale `issues[...]`.
3. `tests/swarm/test_boss_decomposition_depth.py:11-20` — the `_fake_gh_subprocess`
   helper treats *any* non-`[issue, list]` invocation as success-with-empty-stdout.
   That's fine for the tests here, but if a future test depends on the
   idempotency path rejecting, the broad stub could mask it. Minor test-hygiene
   note, not a bug.

**Recommended follow-up:** single small PR replacing the duplicate-coverage
query with a lineage-aware one, plus a note in `_already_maxed_issue_numbers`
about counter cleanup on manual `boss-stuck` removal.

### #6072 — `fix(swarm): parse launchd probe top-level state`

**Scope:** Rewrite `parse_launchd_state` in `scripts/probe_boss_loop_launchd.py`
to track brace depth and only parse fields at block-depth 1 (the top-level
service body). Old logic only skipped `inherited environment` / `default
environment` explicitly and happily accepted `state = active` / `active count
= 1` lines from *any* nested block. New behaviour ignores everything at
depth ≥ 2.

Plus `tests/scripts/test_probe_boss_loop_launchd.py`: add
`SAMPLE_OUTPUT_WITH_NESTED_ACTIVE_COALITIONS` and
`test_parse_launchd_state_ignores_nested_coalition_state`.

**Correctness: PASS.**

Traced the new parser against `SAMPLE_OUTPUT` line-by-line:

- Header `gui/501/... = {` is not matched as a `}` line; field parse is
  skipped because `entered_service` is still False; `opens=1` flips
  `entered_service` to True and lifts `block_depth` to 1.
- Top-level kv lines (`state = spawn scheduled`, `runs = 224`, etc.) are
  captured at depth 1.
- `arguments = {`, `inherited environment = {`, `default environment = {`,
  `environment = {`, and the new `resource coalition = {` / `jetsam
  coalition = {` all lift depth to 2; their internal `state = active` /
  `active count = 1` are skipped by the `block_depth == 1` guard.
- Secret redaction still works (`inherited environment` contents are at
  depth ≥ 2, so `OPENAI_API_KEY => sk-...` lines are never parsed).
- Closing `}` at depth 1 breaks the loop, so any trailing content in the
  same stream is ignored.

**Regression risk:** Low. The previous explicit `skip_block_depth` logic only
handled two environment blocks; the new logic handles *all* nested blocks
uniformly. The only way to break an existing consumer would be if someone
relied on pulling a field from a nested block, which the previous code did not
support either (it was just accidentally right about specific skip points).

**Silent-failure hazards:**

- The `try: fields[...] = int(value) except ValueError: pass` in the
  refactor (`scripts/probe_boss_loop_launchd.py:108-114`) now silently swallows
  bad integers instead of `continue`-ing after the regex match. In the old
  form, `continue` would skip the remainder of the loop body (there was none
  after), so behaviourally this is identical. No new hazard.
- If the whole service body is missing (e.g., launchctl returns a
  surface-level error that is not `Could not find service`), the parser
  returns `None`. Caller handles `None` by reporting "not loaded or
  unreadable". OK.

**Test coverage adequacy:** Good, behavioural (not implementation). The test
checks the *effect*: `state.state == "spawn scheduled"`, `active_count == 0`,
`is_running is False`, `is_spawn_scheduled is True`. That's exactly the
property the probe was lying about before. Existing redaction and
`None`-on-unloaded tests still pass.

What is not exercised:

- A service block nested inside a domain block (e.g., `launchctl print
  gui/501` rather than `gui/501/label`). In that case `entered_service`
  flips at the domain level, and real service fields live at depth ≥ 2,
  which the parser now ignores — returning `None`. This is a non-bug for
  the current call site (`read_launchd_state` prints the specific label),
  but worth a one-line test asserting the parser is *only* valid for
  single-service output.
- A line that contains both `{` and a key/value (e.g., `foo = bar { baz }`)
  — not seen in launchctl output, but the brace-counting logic would
  double-step. Very unlikely to occur in practice.

**Issues found:** none blocking.

**Recommended follow-up:** none. Optionally tighten docstring to say
"requires launchctl print gui/UID/label output; domain-level output is not
supported."

### #6070 — `test(proof-first): cover benchmark publication gate`

**Scope:** Four new tests in `tests/scripts/test_run_proof_first_shift.py`
against `should_trigger_benchmark_rerun` and `has_open_benchmark_publication_pr`.
No production-code change.

**Correctness: PASS (for what it tests).**

The four added tests cover three *don't-trigger* paths and one *branch-name
matcher*:

- `awaiting_first_run_visibility` via the `-1` sentinel
  (`scripts/run_proof_first_shift.py:413`).
- `awaiting_new_benchmark_run` when the latest visible run is the one we
  already triggered (`scripts/run_proof_first_shift.py:422-425`).
- `benchmark_pr_open` — gate returns False when a pub PR is open
  (`scripts/run_proof_first_shift.py:408-409`).
- `has_open_benchmark_publication_pr` correctly requires the prefix
  `benchmark-truth-publication/` with the trailing slash — not `codex/...`,
  not `benchmark-truth-publication-fix`.

**Stage-gate honesty:** Borderline **honest** but **incomplete**. The tests do
validate that the gate *refuses to fire* in three distinct reasons, which is
the right direction for a gate's correctness (a gate that over-fires is worse
than one that under-fires). However:

- The only *True* case exercised by the full test file is the pre-existing
  `test_should_trigger_benchmark_rerun_when_latest_run_is_stale`. The new
  tests do not cover `retry_after_failed_run`
  (`scripts/run_proof_first_shift.py:427-428`) at all. A malformed
  `conclusion` handling (e.g., `"FAILURE"` case-sensitivity) would slip
  past.
- The test `test_should_trigger_benchmark_rerun_waits_for_new_run_after_trigger`
  relies on the 2026-04-14 timestamp being older than wall-clock `_utc_now()`
  to hit the age branch. It does not patch `_utc_now`. Stable in CI today,
  fragile if someone ever freezes time or pins the clock.
- Subtle falsy-id edge case: `run_id and run_id == int(last_triggered_run_id
  or 0)` (`scripts/run_proof_first_shift.py:422`) short-circuits when
  `run_id == 0`. If GitHub ever returns `databaseId: 0` for an empty run row
  (unlikely but nothing in the code rejects it), the stale comparison is
  skipped and the gate goes to `retry_after_failed_run` / `fresh_enough`. Not
  covered.

The tests test **behaviour** (trigger/no-trigger + reason string), which is the
right granularity — they are not testing the implementation. The reason
strings are part of the gate's contract and show up in `actions.append(
f"trigger_benchmark:{trigger_reason}")` in the runtime
(`scripts/run_proof_first_shift.py:1048-1049`), so asserting on them is
justified.

**Regression risk:** Zero — tests only.

**Issues found:** none. Noted gaps above for the follow-up.

**Recommended follow-up:** add two small tests — `retry_after_failed_run`
positive path, and `databaseId=0` edge. Patch `_utc_now` rather than relying
on wall-clock drift in the stale-age test.

### #6053 — `fix(swarm): stabilize lane telemetry db path`

**Scope:** In `aragora/swarm/lane_telemetry.py::LaneTelemetryCollector.__init__`,
resolve the default path to absolute
(`aragora/swarm/lane_telemetry.py:97`). Add `if row is None: return 0.0`
defensive guards to `get_success_rate`, `get_human_intervention_rate`, and
`get_merge_yield` (`aragora/swarm/lane_telemetry.py:268-316`). Tighten the
`_aggregate_row` return annotation to drop the impossible
`tuple[Any, ...]` union. Add one regression test for cwd-change survival.

**Correctness: PASS.**

The bug is real: module-level `_LANE_TELEMETRY = LaneTelemetryCollector()`
in five production modules
(`aragora/swarm/boss_loop.py:85`, `aragora/swarm/campaign.py:41`,
`aragora/swarm/reporter.py:26`, `aragora/swarm/supervisor.py:57`,
`aragora/nomic/dev_coordination.py:119`) is instantiated at first import.
`get_default_data_dir()`
(`aragora/persistence/db_config.py`) can return relative paths
(`Path(".nomic")`, `Path("data")`, or a relative env var). Before this fix,
sqlite would open a new empty DB in any subsequent cwd, silently dropping
telemetry. `.resolve()` at construction time binds the absolute path.

The `row is None` guards are narrow but correct — `sqlite3.Cursor.fetchone()`
returns `None` when no rows match, and the old code then blew up on
`int(row["total"] or 0)`. The replacement returns `0.0`, consistent with the
`total == 0` branch already present below each guard. Not a silent-error
swallow (we are not catching an exception); it's a correct empty-window
default.

**Regression risk:** Low. Callers of these three rate accessors previously
never saw `None`-row execution (they would have crashed with TypeError in
`int(None[...])`). Now they get `0.0`, which is what the `0 / 0 → 0.0`
branches right below already return. No caller I can find treats "zero
requests" differently from "query returned no row at all".

One small asymmetry: `get_avg_time_to_pr` / `get_avg_time_to_merge` still rely
on `float(row or 0.0)` via `_aggregate_scalar`, which is fine because
`_aggregate_scalar` has *its own* `if row is None: return 0` guard
(`aragora/swarm/lane_telemetry.py:344-348`). So those methods were already
safe. The PR correctly avoids touching them.

**Silent-failure hazards:** None. The guards replace crashes with the
already-defined "empty window" value, not with a swallow-and-log.

**Test coverage adequacy:** Adequate.
`test_default_db_path_survives_later_cwd_changes` exercises exactly the field
regression (cwd change after collector creation) with a relative
`ARAGORA_DATA_DIR`. It verifies the *behavioural* consequence — a recorded
lane is actually readable via `get_throughput` — rather than just asserting
`db_path` is absolute.

What is not exercised:

- Explicit relative `db_path` arg (e.g.,
  `LaneTelemetryCollector(db_path="lane.db")`) — still suffers the old bug
  because resolution only happens in the default branch. No production
  caller does this today (all five production sites use no-arg), so not a
  concern, but the fix is *narrower* than "resolve all paths".
- Concurrent writers after cwd-change. Out of scope for this PR.

**Issues found:**

1. `aragora/swarm/lane_telemetry.py:94-97` — the `.resolve()` call is only
   applied to the default-path branch. If a caller ever passes an explicit
   relative `db_path`, they will re-introduce the bug. Cheap guard: also
   resolve in the `else` branch when `db_path` is neither `:memory:` nor
   absolute.

**Recommended follow-up:** optional one-liner to resolve explicit relative
paths too. Low priority.

### #6054 — `chore(governance): reset contract drift ratchet date`

**Scope:** One field in `scripts/baselines/contract_drift_program.json`:
`start_date` moves from `2026-04-09` to `2026-04-17`. Baseline total (655),
weekly reduction (10%), and grace weeks (0) are unchanged.

**Correctness: N/A — policy data change, not code.**

The PR body states the intent clearly: "unblock active contract-repair PRs
from failing a target the repo has not yet earned" and confirms via
`python3 scripts/check_contract_drift_ratchet.py --strict --as-of 2026-04-24`
that the next week's target still bites. I ran the math: with 10% weekly
reduction from 655 and a new start date of 2026-04-17, by 2026-04-24 the
target is ~590 items. So the ratchet still pressures burndown of ~65 items
per week.

**Regression risk:** None to code. The risk is *governance drift*. If every
time the ratchet is about to fail, the start date is bumped to today, the
ratchet has no enforcement value — it silently converts from a deadline to a
"reset button". The commit is defensible as a one-time grace, but the repo
has no record that this is a one-time event, and there is no committed
commitment-device (e.g., "next reset requires N reviewers" / "baseline must
also move down").

**Silent-failure hazards:** The ratchet check script is a CI lane. If future
authors keep resetting `start_date` whenever the target bites, CI stays green
while the underlying backlog does not shrink. This is exactly the class of
hazard the ratchet was introduced to prevent.

**Test coverage adequacy:** N/A (config file). The PR body reports both
`--as-of 2026-04-17` (passes) and `--as-of 2026-04-24` (still fails unless
drift burns down by 65) — good that the author verified both sides.

**Issues found:**

1. `scripts/baselines/contract_drift_program.json:2` — no commit trailer or
   adjacent doc note establishing this as a *one-time* grace. The PR body is
   the only durable evidence, and PR bodies are not policy.

**Recommended follow-up:** add a comment or companion doc (e.g.
`docs/governance/contract_drift_program.md`) stating (a) when `start_date`
can be reset, (b) that `start_total_items` must also move down in the same
PR for any subsequent reset, and (c) who must approve.

## Cross-cutting findings

1. **Shared theme: fail-closed substrate hygiene.** Four of the five PRs
   (everything except #6054) convert silent-pass substrate states into
   truthful failures: the launchd probe no longer reports `ok` via nested
   coalition state (#6072); the boss loop no longer re-enters decomposition
   infinitely and no longer leaks `boss-ready` (#6071); the proof-first gate
   now has explicit "awaiting" reasons covered by tests (#6070); the lane
   telemetry collector no longer silently drops writes after a cwd change
   (#6053). This is a consistent and correct direction for the autonomy
   substrate.

2. **No adversarial interaction between #6071 and #6072.** The launchd probe
   and the decomposer guard are orthogonal — one detects whether the
   boss-loop daemon is actually running, the other governs what the boss
   loop does to individual issues. There is no code path where the two
   new behaviours can combine to produce a bug that neither produces alone.

3. **Decomposer duplicate-coverage now leans entirely on the idempotency
   guard.** The interaction flagged under #6071 (duplicate-coverage query
   still filters by `--label boss-ready`, but children no longer have that
   label) means the function's own de-duplication is only correct while the
   guard holds. If any future change bypasses the guard (human removal of
   `boss-stuck`, multi-host deployment, counter reset), duplicates become
   possible. This is the only cross-PR hazard worth a follow-up.

4. **Testing style is uniformly behavioural, not
   implementation-coupled.** The idempotency guard test asserts
   `mock_subprocess.call_count == 0` (behaviour: no gh calls). The queue-leak
   test asserts `"boss-ready" not in cmd` (behaviour: no label arg). The
   launchd probe test asserts `state.state == "spawn scheduled"` (behaviour,
   not brace-counting internals). The telemetry test asserts a
   `record_lane → get_throughput` round-trip after cwd change. The proof-first
   tests assert returned `(trigger, reason)` tuples. This is a healthy
   pattern across the batch.

5. **Assumptions about launchd state and `boss-ready` are shared across the
   substrate but not centralised.** Both #6072 and the boss-loop layer
   ultimately assume a specific macOS launchd format and a specific label
   vocabulary (`boss-ready`, `boss-stuck`, `boss-quarantined`,
   `boss-invalid`). Nothing is broken today, but if the repo ever goes
   multi-host (non-macOS CI) or adds a second label namespace, both the probe
   and the decomposer will need to be touched in lock-step. Worth a
   separate design note (not a PR) when Foreman reliability firms up.

6. **Governance-layer change (#6054) is different in kind from the other
   four.** It does not harden substrate behaviour; it relaxes a governance
   deadline. The batch is coherent otherwise; #6054 should not be bundled
   mentally with the substrate fixes when assessing "recent substrate
   improvements" — it is a policy reset.
